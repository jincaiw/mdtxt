## 0. Codex 执行总则

本节优先级最高。Codex 在开发过程中必须遵守以下约束。

### 0.1 仓库与历史保护

1. 在现有 Paperling 仓库内开发，不重新创建仓库。
2. 禁止删除 `.git`、重新执行 `git init`、修改已有远程地址、重写提交历史或强制推送。
3. 不把 MarkEdit 仓库整体复制到 Paperling。
4. 不引入 Swift、AppKit 或 Xcode 工程作为跨平台主实现。
5. 可以参考 MarkEdit 的设计与算法；直接复制 MIT 代码时，必须保留相应版权和许可证声明。
6. 保留 Paperling 的 Apache-2.0 许可证、NOTICE 和已有第三方许可证信息。

### 0.2 开发方式

1. 先阅读当前代码、测试和构建配置，再修改。
2. 优先采用渐进式重构，不进行一次性推倒重写。
3. 每个阶段必须保持应用可启动、可编辑、可保存。
4. 新旧编辑模式并存，实时渲染模式稳定前不得删除源码和分屏模式。
5. 每次修改必须同步补充测试，不允许只实现 UI 演示。
6. 不以正则表达式作为 Markdown 结构识别核心；结构识别必须基于 Lezer 语法树。
7. 不在 React 组件中堆积编辑器底层逻辑；CodeMirror 行为应放入独立扩展模块。
8. 不在前端保存明文 AI 密钥；沿用系统钥匙串。
9. 不在主线程执行明显耗时的目录扫描、全文搜索、导出、AI 请求或大型图表渲染。
10. 不为了功能数量牺牲输入稳定性、撤销重做、光标位置和文档数据安全。
11. 不得在一次任务中尝试完成整份 PRD；必须按阶段和独立需求编号迭代，每次保持主分支可构建。
12. 修改跨模块架构前先提交简短 ADR，说明问题、备选方案、选择理由、迁移步骤和回滚方案。
13. 遇到测试失败、数据安全风险或无法确认的跨平台行为时，停止扩展功能，先修复或记录为明确阻塞项。
14. 不擅自更改产品范围、技术栈、许可证、默认文件格式或删除现有用户功能。
15. 所有新增用户可见文案必须通过国际化资源键提供，不得直接硬编码中文或英文。
16. 新增页面、菜单、命令、错误、提示、快捷键说明和帮助内容时，必须同时提交完整简体中文和英文翻译。
17. 国际化资源缺失、变量不匹配、复数规则错误或英文回退泄漏到默认中文界面，视为测试失败。
18. 产品对外名称统一为小写 `mdtxt`；句首、标题、安装包和系统菜单中也不改写为 `Mdtxt` 或 `MDTXT`，除非平台技术限制。
19. 当前产品版本统一为 `0.1.0`；前端包、Tauri、Cargo、构建元数据、关于页面和更新清单必须一致。
20. 品牌迁移必须独立提交，不得与编辑器架构重构、依赖升级或功能新增混在同一提交中。
21. 不直接删除旧配置和旧会话目录；必须先提供一次性迁移、幂等测试和失败回退。
22. 修改应用标识、更新地址或签名配置前必须提交 ADR，并明确开发标识、正式标识和发布密钥的管理方式。

### 0.3 每个阶段的最低交付

Codex 完成一个阶段前，必须执行并记录：

```bash
bun install --frozen-lockfile
bun run test
bun run build
cargo fmt --check --manifest-path src-tauri/Cargo.toml
cargo clippy --manifest-path src-tauri/Cargo.toml --all-targets -- -D warnings
cargo test --manifest-path src-tauri/Cargo.toml
cargo check --manifest-path src-tauri/Cargo.toml
```

涉及 Tauri 配置、Rust 命令、资源、打包或平台能力的阶段，还必须在当前平台执行：

```bash
bun run tauri build --debug
```

只有发布阶段才强制生成正式 Release 安装包和更新产物。CI 使用 macOS、Windows、Linux 矩阵验证；单个平台开发环境不得伪造其他平台已通过。

如果当前操作系统无法构建所有平台安装包，至少完成当前平台构建，并确保 GitHub Actions 覆盖其余平台。

每个阶段同时提交：

- 代码变更；
- 单元测试；
- 必要的集成测试；
- 手工验证清单；
- 架构或用户文档更新；
- 已知问题说明。

### 0.4 依赖与构建约束

1. JavaScript 包管理器统一使用 Bun，提交并维护锁文件；CI 必须使用冻结锁文件安装。基线阶段应通过 `packageManager`、`.tool-versions`、`rust-toolchain.toml` 或等效文件固定 Bun、Node 兼容版本和 Rust toolchain。
2. Rust 依赖通过 Cargo 管理；新增依赖前检查维护状态、许可证、平台支持、二进制体积和安全风险。
3. 不同时引入多个承担相同职责的 Markdown 解析器、状态管理库或 UI 组件库。
4. 不引入 Electron、内置 Chromium 或大型常驻服务来解决单个平台问题。
5. 依赖升级与功能重构分开提交；重大版本升级必须单独验证。
6. 正式构建不得包含调试 MCP Bridge、开发服务器地址、测试密钥或宽松 CSP。

# 1. 项目背景与技术决策

## 1.1 当前 Paperling 的基础

Paperling 当前已经具备较完整的跨平台桌面产品框架：

- Tauri 2 + Rust 桌面外壳；
- React + TypeScript + Tailwind CSS 前端；
- CodeMirror 6 源码编辑器；
- `react-markdown`、remark、rehype 阅读和预览链路；
- macOS、Windows、Linux 构建与文件关联；
- 多标签页、文件树、大纲、全局搜索、命令面板；
- 自动保存、外部文件变化检测、会话恢复；
- KaTeX、mhchem、Mermaid；
- AI 对话、选择区改写、整篇修改建议和行内差异审阅；
- PDF、HTML、DOCX 等导出基础；
- 系统钥匙串、自动更新、单实例和双击文件打开。

这些能力与完整 Typora 类产品所需的“桌面外壳和周边工作流”高度重合，因此应继续复用。

## 1.2 当前主要缺口

Paperling 当前核心交互仍偏向：

```text
Markdown 源码编辑器
    +
独立 HTML 阅读/预览区
```

这与 Typora 的关键体验存在差距：

- 不能在同一个编辑区中直接看到接近最终排版的内容；
- Markdown 标记不会根据光标焦点动态隐藏与显示；
- 图片、公式、表格、图表和任务框主要依赖独立预览；
- 编辑与渲染存在两套 DOM 和两套定位关系；
- 分屏滚动同步增加复杂度，仍无法提供真正的所见即所得体验。

## 1.3 MarkEdit 的参考价值

MarkEdit 使用 CodeMirror 6、Lezer 语法树、Decoration、Widget 和模块化扩展，对标题、链接、任务列表、代码块、表格、数学公式、Mermaid、不可见字符和输入行为进行编辑区内处理。

本项目应吸收以下设计：

- 编辑器核心独立于应用 UI；
- 基础扩展、Markdown 扩展、渲染扩展、交互扩展分层；
- 通过 Lezer 节点定位 Markdown 结构；
- 只处理可视区域和受影响范围；
- 使用 CodeMirror transaction mapping 保持光标、选择区和撤销历史；
- 处理输入法、拼写检查、自动纠错和 WebView 差异；
- 对复杂块采用 Widget 或替换视图，对当前编辑块恢复源码。

## 1.4 最终技术路线

采用以下组合，而不是二选一式重写：

```text
Paperling 产品与跨平台底座
├── Tauri 2
├── Rust
├── React / TypeScript
├── 文件、标签页、窗口、导出、AI、更新
└── macOS / Windows / Linux 打包

新的编辑器核心
├── CodeMirror 6
├── Lezer Markdown Syntax Tree
├── Decoration / WidgetType / ViewPlugin / StateField
├── 单编辑区实时渲染
└── 参考 MarkEdit 的模块组织和性能策略
```

# 2. 产品定义

## 2.1 产品定位

一款轻量、快速、隐私友好、默认使用简体中文、同时完整支持英语、无需建立 Vault 即可使用的跨平台 Markdown 编辑器。

用户可以像使用普通文档编辑器一样直接打开任意 `.md` 或 `.markdown` 文件，在单一编辑区中完成阅读和编辑；同时为专业用户保留源码模式、分屏模式和完整 Markdown 控制能力。

## 2.2 一句话描述

> 打开 Markdown 文件即可获得接近 Typora 的实时编辑体验，同时保留源码、分屏、文件管理、导出和可选 AI 辅助。

## 2.3 产品原则

1. **文件优先**：Markdown 文件是唯一事实来源，不建立专有数据库格式。
2. **本地优先**：默认离线可用，编辑、保存、预览和导出不依赖云服务。
3. **数据安全优先**：任何渲染功能不得改变原始 Markdown 语义。
4. **编辑优先**：输入稳定性高于动画和视觉效果。
5. **跨平台一致**：核心功能和文档结果一致，平台交互符合各系统习惯。
6. **高级能力可退出**：实时模式异常时可立即切换源码模式。
7. **不过度产品化**：不做知识图谱、数据库、白板、团队协作平台或强制账户系统。
8. **可扩展但不臃肿**：V1 建立内部扩展注册机制，不急于上线开放插件市场。
9. **中文优先、双语完整**：简体中文是默认产品语言；英语不是附属翻译，两种语言必须覆盖同一功能范围。
10. **语言不影响文档**：切换产品界面语言不得修改 Markdown 内容、文件名、Frontmatter、导出正文或用户自定义术语。

## 2.4 目标用户

### 主要用户

- 使用 Markdown 写文档、博客、说明书和技术文章的个人用户；
- 软件开发者、运维工程师、产品经理、研究人员和学生；
- 希望替代 Typora，但重视开源、跨平台和本地隐私的用户；
- 经常直接打开单个 Markdown 文件，而不希望先创建工作区的用户。

### 次要用户

- 使用文件夹管理大量 Markdown 文档的用户；
- 需要数学、化学、Mermaid 和代码块的专业写作者；
- 需要 AI 改写和差异审阅，但希望自己控制模型端点的用户。

# 3. 成功指标

## 3.1 核心体验指标

1. 新用户在首次启动后 30 秒内完成打开、编辑和保存 Markdown 文件。
2. 默认实时模式下，常见 Markdown 文档无需查看源码即可完成主要编辑。
3. 光标进入元素后能可靠显示必要 Markdown 标记；离开后恢复渲染状态。
4. 实时模式、源码模式和分屏模式之间切换不改变文档内容和撤销历史语义。
5. 异常退出后，已保存文件不损坏，未保存内容能恢复或明确提示。

## 3.2 性能指标

基准设备定义为：近五年主流 8 核桌面 CPU、16 GB 内存、SSD、集成或入门独立显卡；使用 Release 构建并关闭开发工具。性能报告必须记录操作系统、CPU、内存、文件特征和测量方法。

| 指标 | 目标 |
|---|---:|
| 冷启动到欢迎页可交互 | P50 ≤ 1.5 秒；P95 ≤ 2.5 秒 |
| 已关联 Markdown 文件双击到可编辑 | P95 ≤ 3 秒 |
| 1 MB 普通文档首次打开 Live | P95 ≤ 1.5 秒 |
| 10 MB 文档首次打开 Source | P95 ≤ 3 秒 |
| 10 MB 文档首次打开 Live | P95 ≤ 5 秒；复杂 Widget 延迟加载 |
| 1 MB 文档普通输入延迟 | P95 ≤ 16 ms |
| 10 MB Source 模式输入延迟 | P95 ≤ 32 ms |
| 快速连续输入不丢字符 | 100% |
| 10000 行普通文档滚动 | 无持续卡死，P95 帧耗时建议 ≤ 33 ms |
| 搜索 1000 个 Markdown 文件 | 首批结果 P95 ≤ 1 秒 |
| 自动保存 | 不阻塞编辑线程，不导致可感知光标停顿 |
| 欢迎页空闲内存 | 建议 ≤ 180 MB，作为软目标而非发布阻塞项 |

说明：

- 复杂 Mermaid、大量超大图片和超长公式允许异步、按视口和延迟渲染；
- 性能指标不得通过关闭数据安全、撤销历史或语法正确性达成；
- 超大文件允许自动进入“受限 Live”或提示 Source 模式，但必须保持全文可编辑。

## 3.3 非功能优先级

发生冲突时，按以下顺序决策：

1. 文档数据完整性；
2. 输入法、光标、选择区和撤销重做正确性；
3. 打开、保存、恢复和外部冲突安全；
4. Markdown 语义兼容；
5. 编辑性能；
6. 实时渲染完整度；
7. 视觉效果和动画。

## 3.4 国际化成功指标

| 指标 | 目标 |
|---|---:|
| 简体中文界面覆盖率 | 100% |
| 英文界面覆盖率 | 100% |
| 缺失翻译键 | 0 |
| 中英文变量占位符不一致 | 0 |
| 默认中文界面出现未授权英文回退 | 0 |

## 3.5 质量指标

- 不允许出现可复现的数据丢失缺陷；
- 核心 Markdown 编辑测试通过率 100%；
- 跨平台关键流程自动化测试通过；

- Windows 和 macOS 正式发布必须完成代码签名；
- Linux 提供 AppImage、DEB，RPM 可作为增强项。

# 4. 产品范围与优先级

## 4.1 P0：必须完成

- 单编辑区实时渲染模式；
- 源码模式；
- 分屏模式；
- 阅读模式；
- 文件打开、新建、保存、另存为；

# 5. Markdown 兼容与数据完整性

## FR-COMPAT-001 唯一事实来源

- Markdown 源文本是唯一事实来源；
- CodeMirror `EditorState.doc` 是活动文档的编辑主状态；
- Widget、预览 DOM、React 状态、导出 HTML 和 AI 建议均不是文档主状态；
- 所有修改最终必须形成可审计、可撤销的 CodeMirror transaction。

## FR-COMPAT-002 无编辑往返

打开文件后，如果用户没有修改正文，保存不得无故改变：

- BOM；
- LF/CRLF；
- 尾随换行；
- 未知 Markdown 语法；
- 空白、缩进和表格格式；
- Frontmatter 字段顺序；
- HTML 和自定义扩展内容。

必须建立字节级 round-trip 测试。确需规范化时，只能由用户执行明确的格式化命令。

## FR-COMPAT-003 编辑兼容

- 编辑已知节点时，只修改目标范围；
- 不因切换模式、主题、字体、预览或导出而改写 Markdown；
- 对无法识别或无法安全渲染的内容，始终回退显示源码；
- 不自动把标准 Markdown 转换为产品私有语法；
- 不自动重排列表编号、表格、Frontmatter 或引用格式，除非用户执行相应命令。

## FR-COMPAT-004 保存一致性

- 每次保存关联文档版本号、最后已知磁盘版本和内容哈希；
- 保存前检测外部修改；
- 原子替换失败时不得破坏原文件；
- 保存成功后再更新 dirty、mtime、hash 和恢复数据状态；
- 保存失败必须保留编辑缓冲区和撤销历史。

# 7. 编辑模式

## FR-EDIT-001 实时渲染模式 Live

实时模式是产品目标默认模式。只有在阶段 4 的质量门槛全部通过后才能设为默认；此前 Source 仍为默认或保留 Live Beta 标识。

### 基本行为

1. 同一个 CodeMirror 编辑区同时承担文本编辑与视觉渲染。
2. 未聚焦元素尽可能显示最终视觉效果。
3. 光标或选择区进入元素时，显示完成编辑所需的 Markdown 标记。
4. 不得通过把 Markdown 转换成 HTML 再回写的方式编辑。
5. Markdown 源文本始终保存在 CodeMirror `EditorState` 中。
6. Decoration 和 Widget 只负责显示，不得成为新的事实来源。

### 焦点显示规则

- 当前光标所在行或节点显示必要标记；
- 跨节点选择时显示选择范围内相关标记；
- 多光标时，每个光标附近均应保持可编辑；
- 输入法组合期间不得隐藏正在组合的标记或重建 DOM；
- 鼠标悬停不能替代光标焦点，不应导致编辑区频繁抖动；
- 搜索命中和拼写标记应与 Markdown 装饰共存。

## FR-EDIT-002 源码模式 Source

- 显示完整 Markdown 源码；
- 支持行号、代码折叠、语法高亮；
- 支持多光标、矩形选择、查找替换；
- 作为异常恢复和专业编辑模式长期保留；
- 与实时模式共享同一个 `DocumentSession` 和 CodeMirror 文档，不维护竞争性文本副本。

## FR-EDIT-003 分屏模式 Split

- 左侧源码，右侧阅读预览；
- 支持双向滚动同步；
- 可拖动分隔条；
- 预览中的任务框、链接和标题导航可交互；
- 分屏模式作为复杂语法校验和迁移过渡模式保留。

## FR-EDIT-004 阅读模式 Reader

- 只显示渲染结果；
- 支持文本选择、复制、链接、图片放大和文内跳转；
- 支持阅读模式查找；
- 不显示编辑光标；
- 可快速返回实时或源码模式。

## FR-EDIT-005 模式切换

- 快捷键和工具栏均可切换；
- 切换后保持光标语义位置、选择区和滚动位置；
- 切换不得触发文档保存或内容变更；
- 当前模式按用户偏好持久化；
- 针对超大文档可以提示关闭复杂实时渲染，但不能强制丢弃编辑内容。

# 8. Markdown 实时渲染详细要求

## 8.1 内联元素

### FR-MD-001 粗体、斜体、删除线

- 非编辑状态隐藏 `**`、`*`、`_`、`~~` 等标记；
- 光标进入后恢复标记；
- 嵌套语法正确处理；
- 不因样式改变导致光标跳动；
- 快捷键支持包裹选区或插入成对标记。

### FR-MD-002 行内代码

- 使用等宽字体和背景；
- 未聚焦时可隐藏反引号；
- 聚焦时恢复完整源码；
- 多反引号边界必须正确。

### FR-MD-003 链接

- 未聚焦时显示链接文本，弱化或隐藏 URL；
- 聚焦后显示完整 `[text](url "title")`；
- 支持 Ctrl/Cmd + 点击打开；
- 默认阻止危险协议；
- 支持编辑链接、复制链接和在系统浏览器打开。

### FR-MD-004 图片

- 未聚焦时在编辑区显示图片；
- 聚焦图片语法时显示 Markdown 源码和图片操作条；
- 支持相对路径、绝对路径、`file://` 受控读取和远程 HTTPS 图片；
- 支持替代文本、标题、复制路径、在文件夹中显示；
- 图片加载失败时显示可编辑错误占位，不隐藏源码；
- 大图按编辑区宽度缩放，不阻塞主线程；
- 禁止自动上传图片。

### FR-MD-005 自动链接与邮箱

- 正确显示 GFM 自动链接；
- 点击前进行协议校验；
- 不应在用户输入过程中自动重写文本。

## 8.2 块级元素

### FR-MD-010 标题

- H1–H6 显示差异化字号和层级；
- 光标进入时显示 `#` 或 Setext 标记；
- 标题左侧可提供折叠和锚点操作；
- 标题变化实时同步大纲；
- 标题字号变化不得破坏滚动锚点。

### FR-MD-011 段落与换行

- 正确区分软换行、硬换行和段落；
- 保留原始 LF/CRLF；
- 可配置段间距、行高和最大内容宽度；
- 不自动更改用户的换行风格。

### FR-MD-012 引用

- 显示左侧引用线和缩进；
- 支持多级引用；
- 聚焦时显示 `>`；
- 回车继续引用，空引用行退出引用。

### FR-MD-013 列表

- 支持无序、有序和嵌套列表；
- 回车自动续号；
- 空列表项回车退出；
- Tab/Shift+Tab 调整层级；
- 有序列表是否自动重排可配置；
- 不得错误修改代码块中的列表符号。

### FR-MD-014 任务列表

- 显示可点击复选框；
- 点击必须通过 CodeMirror transaction 修改 `[ ]`/`[x]`；
- 保留选择区和撤销历史；
- 支持键盘切换；
- 已完成文本样式可配置。

### FR-MD-015 分隔线

- 未聚焦时显示水平线；
- 聚焦后显示原始 `---`、`***` 或 `___`；
- 与 YAML Frontmatter 正确区分。

### FR-MD-016 代码块

- 支持围栏代码块和缩进代码块；
- 显示语言、复制按钮、折叠按钮；
- 支持主流语言语法高亮；
- 当前编辑代码块必须保持纯文本输入稳定；
- 长代码块按需高亮；
- 代码块内部禁用 Markdown 行内渲染；
- 可配置显示行号；
- 不执行代码。

### FR-MD-017 表格

V1 采用“源码保持 + 视觉增强”的渐进方案，不直接构建复杂电子表格模型。

必须支持：

- GFM 表格渲染；
- 光标位于表格时出现浮动工具栏；
- 添加、删除、移动行列；
- 设置左、中、右对齐；
- Tab/Shift+Tab 在单元格之间移动；
- 最后一个单元格按 Tab 新增一行；
- 一键整理 Markdown 表格格式；
- 单元格内转义管道符；
- 表格过宽时水平滚动。

复杂可视化单元格编辑可在后续阶段实现，但底层必须始终能无损还原 Markdown。

### FR-MD-018 数学公式

- 支持 `$...$` 和 `$$...$$`；
- 使用 KaTeX；
- 支持 mhchem；
- 未聚焦时渲染，聚焦时显示源码或源码与预览组合；
- 渲染错误显示错误信息，不修改源码；
- 公式渲染按需加载和缓存；
- 对可见区域之外的公式延迟渲染。

### FR-MD-019 Mermaid

- 支持常用 Mermaid 图表；
- 代码块标记为 `mermaid` 时显示图表预览；
- 当前块聚焦时优先保证源码编辑；
- 支持源码/图表切换；
- 图表异步渲染、可取消、可缓存；
- 设置超时与复杂度限制；
- Mermaid 输出必须经过安全配置，禁止任意脚本。

### FR-MD-020 Frontmatter

- 支持 YAML Frontmatter；
- 源码模式完整显示；
- 实时模式可折叠成属性卡；
- 属性卡修改通过 transaction 回写 YAML；
- 不支持的复杂 YAML 结构退回源码编辑；
- 不擅自排序或改写用户字段。

### FR-MD-021 脚注

- 支持 GFM/常见 Markdown 脚注格式；
- 文中脚注引用可点击跳转；
- 编辑时可查看定义；
- 导出保持一致。

### FR-MD-022 HTML

- 源码模式显示原始 HTML；
- 阅读和导出时使用白名单清洗；
- 实时模式仅渲染安全子集；
- 禁止脚本、iframe、事件属性和危险 URL；
- 无法安全渲染的 HTML 显示源码占位。

### FR-MD-023 Callout

- 支持项目现有约定或明确选定的一种语法；
- 不创造难以迁移的私有格式；
- 推荐兼容 GitHub/常见引用式提示块；
- 导出样式一致。

### FR-MD-024 Wikilink

- 支持 `[[file]]` 和可选别名；
- 默认在当前文件所在目录及已打开文件夹内解析；
- 提供自动补全；
- 不要求建立 Vault；
- 找不到目标时显示明确状态，不自动创建文件，除非用户执行创建命令。

# 9. 编辑操作与输入体验

## FR-INPUT-001 输入法

必须专项验证：

- macOS 中文拼音；
- Windows 微软拼音；
- Linux IBus/Fcitx；
- 候选窗口位置；
- 组合期间撤销、回车、空格和方向键行为。

组合输入期间禁止：

- 重建整个 EditorView；
- 替换当前组合范围 DOM；
- 自动隐藏当前语法标记；
- 触发全量 React state 回写。

## FR-INPUT-002 撤销与重做

- 所有编辑操作进入统一 CodeMirror 历史；
- 表格、任务框、属性卡、图片操作和 AI 接受操作均可撤销；
- 标签页切换必须隔离历史；
- 模式切换不得清空历史；
- 外部文件重载前必须明确提示历史处理策略。

## FR-INPUT-003 复制粘贴

- 纯文本按原文插入；
- URL 粘贴到选区时可转成链接；
- HTML 富文本可转换为 Markdown；
- TSV 可转换为 GFM 表格；
- 图片粘贴保存到可配置资源目录；
- 文件未保存时提示先保存或选择资源位置；
- 粘贴转换失败时安全降级为纯文本。

## FR-INPUT-004 拖放

- 支持拖入 Markdown 文件打开；
- 支持拖入图片插入；
- 支持拖入其他文件生成相对链接；
- 多文件拖入时给出明确行为；
- 禁止无提示覆盖已有资源。

## FR-INPUT-005 Slash 命令

行首或空白位置输入 `/` 时可调用：

- 标题；
- 列表；
- 任务；
- 引用；
- 表格；
- 代码块；
- 数学公式；
- Mermaid；
- 图片；
- 链接；
- Callout；
- Frontmatter。

Slash 菜单必须支持键盘导航、搜索和 Esc 关闭。

## FR-INPUT-006 快捷键

- 使用统一命令注册表；
- 展示平台对应键位；
- macOS 使用 Cmd，Windows/Linux 使用 Ctrl；
- 避免 WebView 保留快捷键冲突；
- 支持用户自定义，并检测冲突；
- 菜单、命令面板和快捷键调用同一个命令实现。

# 10. 文件、文件夹和标签页

## FR-FILE-001 文件格式

- 支持 `.md`、`.markdown`；
- 可选支持 `.mdown`、`.mkd`；
- 默认 UTF-8；
- 识别 UTF-8 BOM；
- 对其他编码给出检测和转换提示；
- 保留 LF/CRLF；
- 保存采用临时文件 + 原子替换策略，失败时保留原文件。

## FR-FILE-002 新建、打开、保存

- 新建未命名文档；
- 打开单个或多个文件；
- 另存为；
- 保存全部；
- 双击系统关联文件打开；
- 单实例接收后续打开请求；
- 保存失败显示可操作错误，不关闭文档。

## FR-FILE-003 多标签页

- 支持固定、关闭、关闭其他、关闭右侧；
- 显示未保存状态；
- 支持拖动排序；
- 切换时保存各自滚动、选择区和模式；
- 同一路径默认只打开一个标签；
- 文件移动或删除时显示状态；
- 关闭脏标签必须 Save/Discard/Cancel。

## FR-FILE-004 文件夹工作区

工作区只是已打开文件夹，不引入专有 Vault 概念。

功能包括：

- 文件树；
- 新建、重命名、移动、删除文件和文件夹；
- 全局搜索；
- 文件快速打开；
- Wikilink 解析；
- 最近文件夹；
- 忽略规则；
- 展开状态保存。

删除默认移入系统废纸篓/回收站；无法使用系统回收站时必须二次确认。

## FR-FILE-005 外部变化

- 监测当前文件和打开文件夹变化；
- 未修改文档可自动重载并尽量保持位置；
- 已修改文档出现冲突对话框；
- 提供比较、重新加载、保留本地、另存为；
- 不允许静默覆盖外部修改。

## FR-FILE-006 自动保存与恢复

- 自动保存可关闭；
- 采用防抖策略；
- 仅保存已有路径文件；
- 未命名文档写入恢复存储，不自动决定正式路径；
- 异常退出后恢复标签、内容和光标；
- 正常关闭后清理无用恢复数据；
- 恢复数据不得包含 AI 密钥。

# 11. 导航、搜索与大纲

## FR-NAV-001 文档大纲

- 从语法树提取 H1–H6；
- 点击跳转；
- 跟随光标高亮；
- 支持折叠层级；
- 标题改变后增量更新；

# 16. 跨平台要求

## 15.1 通用原则

- 三平台均提供完整简体中文和英文界面；
- 应用首次安装默认中文，不因系统语言不同而改变；
- 平台原生菜单遵循对应系统术语和位置；
- 同一功能使用稳定动作 ID，显示名称按语言和平台生成；

- 前端编辑器核心保持平台无关；
- 平台差异集中在 `src-tauri` 和小型前端 adapter；
- 不在业务组件中散布大量 `navigator.platform` 判断；
- 建立统一 `PlatformService`；
- 每个平台使用本地 CI Runner 或 GitHub Actions 验证。

## 15.2 macOS

- Apple Silicon 必须支持；Intel 可根据发布成本评估；
- 支持 DMG；
- 正式版完成签名和 notarization；
- Cmd 快捷键；
- 支持系统全屏、触控板滚动、Retina；
- 文件菜单和应用菜单符合 macOS 习惯；
- 系统钥匙串；
- 打印和 PDF 使用 WKWebView/AppKit 桥接；
- V1 以官网签名/notarized 分发为主，不强制 App Sandbox；文件访问仍按最小权限和用户授权设计。若后续进入 Mac App Store，再单独设计 security-scoped bookmark；
- 中文输入法候选窗口位置正确。

## 15.3 Windows

- 支持 Windows 10/11；
- 使用 WebView2；
- MSI 和 NSIS；
- 正式版代码签名；
- Ctrl、Alt 快捷键避免与 WebView2 保留键冲突；
- 支持长路径、盘符、UNC 路径，并覆盖大小写不敏感、保留文件名和文件锁定差异；
- Credential Manager；
- 系统文件关联；
- 高 DPI 和多显示器；
- 防止 Defender/SmartScreen 因构建方式产生不必要误报。

## 15.4 Linux

- 优先支持主流 x86_64 发行版；
- AppImage 和 DEB 为必须，RPM 为建议；
- 兼容主流 WebKitGTK；
- 支持 Wayland 和 X11；
- 钥匙串使用 Secret Service，不可用时明确提示；
- 文件对话框兼容 xdg-desktop-portal；
- 字体缺失时合理回退；
- PDF 导出必须有可靠降级方案；
- 不把某一桌面环境命令写死为唯一实现；
- 工作区搜索默认不跟随指向工作区外部的符号链接，需显式配置。

## 16.3 文档会话与状态边界

每个标签页对应一个独立 `DocumentSession`。会话至少保存：

- 当前 `EditorState`，包括文档、选择区和撤销历史；
- 文件路径、编码、换行符、BOM 和尾随换行状态；
- 当前编辑版本号与最后已保存版本号；
- 最后已知磁盘 mtime、size 和可选内容哈希；
- 当前模式、滚动锚点和恢复 ID。

### CodeMirror 管理

- 文档文本；
- 光标和选择区；
- 撤销重做；
- 编辑器装饰；
- 当前 Markdown 语法树；
- 编辑器内查找状态。

### React 管理

- 标签页和会话元数据；
- 文件树与大纲面板；
- 设置、弹窗、命令面板；
- AI 面板；
- 当前布局；
- 保存状态和通知。

React 不以每次按键产生的全文字符串驱动编辑器。预览、保存、统计、恢复或 AI 需要正文时，通过 `DocumentSession` 获取带 document ID 和 version 的节流快照。

### Rust 管理

- 文件系统访问；
- 原子保存；
- 目录监听和全文搜索；
- 系统钥匙串；
- AI 网络请求；
- 导出与平台能力；
- 单实例和文件关联。

### 标签页与 EditorView 生命周期

- 一个窗口默认只创建一个活动 `EditorView`；
- 每个标签保存自己的 `EditorState`，切换标签时使用 `view.setState(session.editorState)`；
- 切换标签前将活动 View 的最新 state 回存到对应会话；
- Live 与 Source 通过 Compartment 重配置同一会话，不重建文档或撤销历史；
- Split 的右侧是只读预览，不维护第二份可编辑 Markdown；
- Reader 使用带版本号的只读快照；
- 多窗口列为 P2 时，每个窗口独立拥有活动 View 和会话集合。

### 异步结果规则

保存、导出、预览、统计、AI、图片和 Mermaid 等异步操作必须携带：

- document ID；
- 发起时的 document version；
- request ID。

返回时若文档、版本或请求已过期，则丢弃结果，不得覆盖新内容。

### 禁止的状态设计

- 不在 React `useState` 中把每次按键产生的完整文档作为持续主状态；
- 不维护“CodeMirror 文本”和“React 文本”两个持续竞争的主状态；
- 不通过销毁重建 `EditorView` 实现普通配置变化；
- 不在保存、预览、统计或 AI 结果返回时覆盖更新版本的编辑内容；
- 不让未激活标签的全文形成多份无边界复制；超大文档标签必须评估内存占用。

## 16.4 编辑器命令协议

所有编辑能力统一为命令：

```ts
interface EditorCommand<T = unknown> {
  id: string;
  title: string;
  shortcut?: PlatformShortcut;
  when?: (context: EditorContext) => boolean;
  run: (context: EditorContext, args?: T) => boolean | Promise<boolean>;
}
```

命令来源：

- 菜单；
- 工具栏；
- 命令面板；
- 快捷键；
- Slash 菜单；
- 自动化测试。

禁止为同一功能维护多套实现。

## 16.5 文档会话接口建议

```ts
interface DocumentSession {
  id: string;
  path: string | null;
  editorState: EditorState;
  version: number;
  savedVersion: number;
  diskRevision?: {

  format: {
```

保存、导出、AI 和预览必须读取明确版本的快照；操作完成时若会话版本已变化，不得用旧结果覆盖新内容。

# 18. 实时渲染技术方案

## 17.1 核心机制

必须使用：

- `syntaxTree(state)` 获取结构；
- `StateField<DecorationSet>` 或 `ViewPlugin` 管理装饰；
- `Decoration.mark` 处理样式；
- `Decoration.replace` 和 `WidgetType` 处理隐藏标记及复杂预览；
- `Decoration.line` 处理块级背景；
- transaction mapping 更新位置；
- `ViewUpdate.changedRanges` 限制重算范围；
- `visibleRanges` 限制复杂 Widget 渲染。

## 17.2 渲染状态

每个节点至少具有：

```text
Source：完整源码
Styled：源码仍存在，仅应用视觉样式
Collapsed：隐藏部分标记
Preview：用 Widget 显示复杂结果
Editing：当前节点恢复可编辑源码
Error：渲染失败，保留源码并显示错误
```

## 17.3 焦点判定

建立统一的 `EditFocusResolver`：

输入：

- 主光标；
- 多选择区；
- IME composition range；
- 鼠标操作目标；
- 当前 Widget 内交互；
- 查找结果。

输出：

- 必须恢复源码的节点范围；
- 可以隐藏标记的节点范围；
- 需要延迟刷新范围。

不同 renderer 不得自行实现互相冲突的焦点判定。

## 17.4 异步 Widget

数学、Mermaid、图片元数据等异步渲染必须：

- 具备请求 ID；
- 文档变更后可取消或丢弃旧结果；
- 不把异步结果直接写入文档；
- 使用内容哈希缓存；
- 限制并发；
- 对不可见节点延迟执行；
- Widget 销毁时释放观察器和事件监听。

## 17.5 大文件降级

阈值不能只依赖文件大小，应综合字节数、行数、最长行、图片数量、数学节点和 Mermaid 节点数量。初始建议阈值为 5 MB 或 100000 行，最终由基准测试调整。

超过阈值时进入“受限 Live”：

- 仅保留标题、强调、列表等低成本装饰；
- 默认关闭不可见区域的复杂 Widget；
- Mermaid、数学和图片仅按视口与用户触发渲染；
- 大图片延迟加载；
- 大纲、统计和搜索增量或后台计算；
- 明确显示当前降级状态和原因；
- 用户可切换 Source，或手动尝试完整 Live；
- 不能静默改变文档内容，也不能自动永久修改用户默认模式。

# 19. 安全与隐私

## SEC-001 文件权限

- Tauri capabilities 采用最小权限；
- 只允许访问用户明确打开的文件和文件夹；
- 防止路径穿越、符号链接逃逸和未经授权的工作区外访问；
- 资源读取限制在文档目录、用户授权目录或明确选择路径；
- 不开放任意命令执行。

## SEC-002 内容安全策略

- 保持严格 CSP；
- 禁止 `eval` 和任意脚本；
- HTML 预览经过 sanitize；
- Mermaid 使用安全等级；
- 外部链接统一通过 opener 和 URL 白名单；
- 禁止 `javascript:`、危险 `data:` 和未知协议。

## SEC-003 网络

- 默认无需网络；
- 仅更新检查、用户允许的远程图片和 AI 可能访问网络；
- 远程图片默认不自动加载，用户可按文档、会话或全局允许；
- 远程图片请求不得携带应用密钥、文档内容或不必要的本地路径信息；
- AI 请求仅发送用户明确范围；
- 不内置未经说明的遥测。

## SEC-004 密钥

- 使用 macOS Keychain、Windows Credential Manager、Linux Secret Service；
- 前端只短暂持有必要值；
- 日志和错误报告进行脱敏；
- 钥匙串不可用时不得静默降级为明文存储。

## SEC-005 更新

- 自动更新包必须签名验证；
- 更新源固定为受控 HTTPS 地址；
- 支持跳过版本；
- 更新失败不影响现有版本启动。

# 20. 可访问性与国际化

## A11Y-001 可访问性基础

- 所有交互控件必须可通过键盘访问；
- 焦点状态清晰可见；
- 图标按钮必须有可本地化的无障碍名称；
- 不只通过颜色传达状态；
- 支持系统高对比度和减少动态效果；
- 编辑器、对话框、文件树、命令面板和设置页具备合理的语义结构；
- 中英文无障碍标签必须与可见文案含义一致；
- 屏幕阅读器不得朗读国际化资源键、Markdown 标记隐藏占位符或调试内容。

## I18N-001 支持语言与默认策略

V1 正式支持：

- 简体中文：`zh-CN`；
- 英语：`en-US`。

默认策略：

1. 首次全新安装默认使用简体中文，不根据操作系统语言自动改为英文；
2. 用户可在首次启动页或设置中选择英语；
3. 用户选择持久化到本地设置，升级后保持；
4. 语言切换默认即时生效，不要求重启应用；
5. 极少数由操作系统控制、无法动态更新的原生菜单或窗口元数据，可提示用户重启后完整生效；
6. 未知语言或损坏配置回退到 `zh-CN`；
7. 单个中文翻译键缺失时回退到 `en-US`，同时在开发环境报警；正式发布前缺失键必须为零；
8. 不根据文档内容语言自动切换应用界面。

## I18N-002 覆盖范围

下列所有用户可见内容必须国际化：

- 欢迎页、首次引导、空状态和教程；
- 标题栏、菜单栏、上下文菜单和托盘菜单；
- 工具栏、状态栏、标签页和侧边栏；
- 文件树、大纲、全局搜索和命令面板；
- 设置、对话框、确认框、Toast 和错误页；
- 编辑器占位符、Slash 命令、表格工具和查找替换；
- 快捷键名称、分类、说明和冲突提示；
- AI 面板、默认操作、模型设置、风险提示和差异审阅；
- 导出、打印、更新、恢复、外部冲突和文件权限提示；
- 安装后内置教程、帮助文档、隐私说明和关于页面；
- Tauri/Rust 返回的业务错误；
- 原生文件菜单和应用菜单中由应用控制的文案；
- 无障碍名称、Tooltip、ARIA 标签和测试可见文本。

以下内容不得自动翻译：

- 用户 Markdown 正文；
- 文件名、目录名和工作区名称；
- 代码、代码语言名和命令行参数；
- 用户自定义主题名、模型名、端点和提示词；
- Frontmatter 键值；
- URL、Wikilink 目标和 Mermaid 源码；
- 用户自行添加的术语和模板。

## I18N-003 资源组织

推荐使用成熟、轻量、支持 TypeScript 的国际化方案，例如 `i18next` + `react-i18next`。最终选择必须通过 ADR 确认，不得自行实现简陋字符串替换系统。

资源规则：

- 相同概念复用同一键，避免同义词漂移；
- 长文本按功能命名空间拆分；
- 不把 Markdown、HTML 或可执行脚本混入普通翻译资源；
- 翻译资源必须纳入版本控制，不依赖运行时在线下载；
- 正式包离线可完整显示两种语言。

## I18N-004 类型安全与资源校验

必须实现：

- 基于主语言资源生成或约束 TypeScript 键类型；
- CI 校验中英文键集合完全一致；
- CI 校验插值变量名称和数量一致；
- CI 校验 ICU 复数、选择和日期格式语法；
- 检测空字符串、重复键和无引用键；
- 检测 React/TSX、Rust 业务层中新增的可疑硬编码用户文案；
- 允许明确白名单：协议关键字、文件扩展名、代码、品牌名和标准缩写；
- 翻译资源解析失败时构建失败，不在运行时静默忽略。

## I18N-005 动态语言切换

切换语言后必须即时更新：

- React 页面和对话框；
- CodeMirror 扩展中的 Tooltip、占位符和命令名称；
- 命令面板索引；
- Slash 命令标签与描述；
- 快捷键帮助；
- 文件状态、统计单位和时间显示；
- AI 默认操作名称；
- 错误提示和 Toast；
- 当前可见设置项和菜单。

切换语言不得：

- 重建活动文档；
- 清空撤销历史；
- 改变光标、选择区或滚动位置；
- 关闭标签页或侧边栏；
- 触发保存；
- 修改文档正文；
- 丢失未保存设置。

## I18N-006 原生菜单与 Rust 错误

前后端边界采用错误代码而不是最终人类文本：

```ts
type AppError = {
  code: string;
  params?: Record<string, string | number>;
  technicalDetail?: string;
};
```

规则：

- Rust 返回稳定错误代码和安全参数；
- 前端根据当前语言映射为用户文案；
- 技术详情仅用于日志或可展开详情，不直接覆盖本地化主提示；
- 未识别错误代码使用通用中文/英文回退文案；
- 原生日志保持技术英文可接受，但用户界面不得直接暴露 Rust Debug 字符串；
- macOS、Windows、Linux 原生菜单由统一菜单模型生成，避免分别硬编码；
- 菜单角色名称遵循平台习惯，例如 macOS 的“关于”“设置”“退出”。

## I18N-007 日期、时间、数字与单位

使用 `Intl` API 或等效标准实现：

- 日期和时间；
- 文件大小；
- 字数、字符数、行数；
- 百分比；
- 相对时间；
- 更新版本时间；
- 搜索结果数量；
- AI Token 或用量显示。

规则：

- `zh-CN` 默认使用中文本地习惯；
- `en-US` 使用英语本地习惯；
- 文件字节计算保持一致，显示层本地化；
- 不手工拼接“个”“项”“files”“minutes”等单位；
- 复数规则必须使用国际化库或 `Intl.PluralRules`；
- 内部存储使用稳定 ISO、整数或枚举，不存储格式化后的文本。

## I18N-008 中文排版与字体

简体中文界面必须：

- 使用系统中文字体栈，避免强制下载大体积中文 Web 字体；
- macOS 优先系统苹方等平台字体；
- Windows 优先微软雅黑或系统 UI 字体；
- Linux 使用系统 `sans-serif` 并验证常见发行版中文字体回退；
- 不因缺少中文字体出现方框、乱码或布局崩溃；
- 按钮和标签预留中文长度；
- 中英文混排时标点、空格和行高视觉自然；
- 中文正文编辑字体与应用 UI 字体设置分离；
- 不自动修改用户 Markdown 中英文标点。

## I18N-009 搜索、命令与排序

- 命令面板可通过中文名称、英文名称、常用别名和快捷键搜索；
- 中文界面下仍可输入英文命令名称找到功能；
- 英文界面下可配置是否保留中文别名，默认保留；
- Slash 命令支持中英文关键词；
- 设置搜索支持中英文同义词；
- 文件名与正文搜索不因 UI 语言改变匹配结果；
- 中文搜索不得依赖空格分词；
- 排序使用 `Intl.Collator` 或稳定平台无关规则；
- 相同目录在三平台和两种语言下应尽量保持可预测顺序；
- 不对用户文件名做拼音转换或自动翻译。

## I18N-010 快捷键与平台术语

快捷键模型将“动作”“按键绑定”“显示文本”分离：

- 动作 ID 稳定，例如 `file.open`, `editor.toggleLive`；
- 键位按平台解析；
- 显示名称通过当前语言资源获取；
- macOS 使用 `⌘`、`⌥`、`⇧` 等平台符号；
- Windows/Linux 使用 `Ctrl`、`Alt`、`Shift`；
- 不把中文动作名写入持久化键位配置；
- 用户自定义快捷键升级和语言切换后保持；
- 冲突提示需明确显示两个本地化动作名称。

## I18N-011 AI 国际化

AI 功能区分：

- 应用 UI 语言；
- 文档主要语言；
- 用户明确要求的输出语言。

默认行为：

- UI 提示随应用语言变化；
- AI 不因应用默认中文而擅自把英文文档翻译为中文；
- “润色”“缩短”“扩写”等操作默认保持选中文本原语言；
- “翻译”操作必须让用户选择目标语言，默认目标与应用语言一致；
- Ask 模式默认使用用户提问语言回复；
- 系统提示词模板需有中英文版本，但必须保持同等安全与功能约束；
- 模型返回的错误通过错误代码或安全映射本地化；
- API 请求不得发送不必要的产品语言、系统语言或设备区域信息。

## I18N-012 导出、打印与元数据

- 导出正文保持原文，不随 UI 语言自动翻译；
- 导出对话框、进度、错误和选项随应用语言；
- HTML 模板中的产品生成器说明可本地化；
- 文档 `lang` 属性优先根据用户导出设置或文档内容推断，不直接等于应用 UI 语言；
- PDF 页眉页脚中的日期、页码说明按导出语言设置；
- DOCX 样式名内部保持稳定，显示名称可本地化；
- 打印系统对话框由操作系统决定语言；
- 导出设置提供“跟随文档”“简体中文”“英语”三个元数据语言选项。

## I18N-013 首次启动与语言设置

首次启动：

- 默认显示简体中文欢迎页；
- 在醒目但不干扰的位置提供 `English` 切换入口；
- 选择语言后立即更新当前页面；
- 不要求创建账户或联网；
- 语言选择在完成引导前也要持久化；
- 首次启动教程分别提供完整中文和英文内容。

设置页：

- 路径：设置 → 外观或通用 → 语言；
- 选项显示为 `简体中文`、`English`，避免用户切换后无法识别；
- 显示“部分系统菜单可能需要重启后完全更新”的条件提示；
- 提供恢复默认语言操作；
- 默认语言值是 `zh-CN`，不使用 `system` 作为 V1 默认。

## I18N-014 后续语言扩展能力

V1 只承诺简体中文和英语，但架构必须允许后续增加其他语言：

- 不在业务逻辑中判断 `language === "zh-CN"` 来改变功能；
- 不假设所有语言从左到右，布局基础组件避免绝对方向命名；
- CSS 优先使用逻辑属性，例如 `margin-inline-start`；
- 为未来 RTL 做结构准备，但 V1 不要求正式支持 RTL；
- 不把中文或英文作为数据库枚举；
- 新语言包可独立增加，不需要修改编辑器核心；
- 语言包版本必须与应用版本匹配，不支持未经签名的远程语言包。

## I18N-015 本地化日志与隐私

- 日志中的稳定事件名、错误代码和技术字段使用英文标识；
- 用户可见日志摘要按当前语言显示；
- 日志不得记录完整 Markdown 正文、AI 密钥和敏感路径；
- 崩溃报告若未来启用，必须明确征得同意；
- 不向第三方翻译服务上传界面资源或用户文档；
- 翻译工作流可以离线或通过代码评审完成。

- 损坏 locale 配置回退到 `zh-CN`；

- E2E 截图检测明显截断、重叠和资源键泄漏；
- 源代码硬编码扫描通过。

## 20.5 输入法专项门槛

IME 行为不能仅依赖 jsdom 或普通浏览器自动化。每个候选发布版必须在真实系统 WebView 中手工验证：

- macOS：系统简体拼音；
- Windows：微软拼音；
- Linux：IBus 或 Fcitx5；
- 组合文本跨粗体、列表、表格和代码块；
- 撤销、回车、空格、方向键和候选框位置。

# 22. 性能工程

## PERF-001 编辑循环

- 每个窗口的活动 CodeMirror View 只创建一次，标签切换采用独立 `EditorState` 替换；
- 不因 React props 变化销毁重建编辑器；
- 输入回调只传最小必要信息；
- 保存快照节流或防抖；
- 不在每次光标移动调用 `doc.toString()`；
- 不在每次按键完整遍历 syntax tree；
- 大纲、统计和预览采用增量或后台计算。

## PERF-002 Decoration

- 只重算变更范围和必要相邻块；
- 大型 Widget 只处理可见范围；
- 复用 Decoration 和 Widget；
- 避免频繁测量 DOM；
- 把读布局和写布局分批执行；
- 监听器在销毁时清理。

## PERF-003 预览

- KaTeX、Mermaid、语言高亮按需加载；
- Mermaid 限制并发；
- 图片懒加载；
- 远程图片失败快速超时；
- 渲染缓存由内容哈希和主题组成 key。

## PERF-004 性能基准

建立可重复脚本，至少输出：

- 启动时间；
- 打开不同大小文件耗时；
- 连续输入 P50/P95；
- Decoration 更新耗时；
- 模式切换耗时；
- 内存占用；
- 全局搜索耗时；
- Mermaid 批量渲染耗时。

# 23. 数据模型

## 22.1 标签页

```ts
interface DocumentTab {
  id: string;
  path: string | null;
  displayName: string;
  encoding: "utf-8" | "utf-8-bom" | string;
  lineEnding: "lf" | "crlf";
  dirty: boolean;
  mode: "live" | "source" | "split" | "reader";
  selection: { anchor: number; head: number };
  scrollAnchor?: DocumentAnchor;
  lastKnownModified?: number;
  recoveryId?: string;
}
```

不得在标签模型中长期保存多个不一致的完整文档副本。非活动标签可保存 CodeMirror state 或受控快照，但切换策略必须经过内存和历史测试。

## 22.2 设置

设置必须版本化：

```ts
interface AppSettingsV1 {
  schemaVersion: 1;
  appearance: AppearanceSettings;
  editor: EditorSettings;
  markdown: MarkdownSettings;
  files: FileSettings;
  shortcuts: ShortcutSettings;
  ai: AISettingsWithoutSecret;
}
```

升级时提供迁移函数，未知字段尽量保留。

## 22.3 国际化状态

```ts
type SupportedLocale = "zh-CN" | "en-US";

interface LocaleSettings {
  locale: SupportedLocale;
  fallbackLocale: "en-US";
  firstRunLanguageChosen: boolean;
}

interface LocalizedCommand {
  id: string;
  nameKey: string;
  descriptionKey?: string;
  aliases: Partial<Record<SupportedLocale, string[]>>;
  defaultKeybinding?: Partial<Record<"macos" | "windows" | "linux", string>>;
}
```

持久化时保存稳定 ID 和 locale 枚举，不保存已经翻译后的动作名称。

## 22.4 恢复数据

- 独立于正式设置；
- 包含未保存缓冲区、标签顺序、路径、模式和位置；
- 定期写入；
- 使用校验和；
- 成功恢复后不立即删除，待正式保存或正常退出后清理；
- 提供恢复失败的备份文件位置。

# 28. 明确禁止事项

1. 禁止把编辑器改为 `contenteditable` 自研富文本内核。
2. 禁止用正则表达式替代 Lezer 语法树进行完整 Markdown 渲染。
3. 禁止每次输入都把全文转 HTML 并替换编辑区。
4. 禁止在 React render 中创建 CodeMirror View。
5. 禁止模式切换时序列化再反序列化 Markdown。
6. 禁止隐藏 Markdown 后无法通过光标恢复源码。
7. 禁止 Widget 持有唯一数据而源码中不存在对应内容。
8. 禁止复杂渲染错误导致文本不可访问。
9. 禁止默认执行 HTML、脚本、代码块或 Mermaid 中的任意代码。
10. 禁止静默覆盖外部修改。
11. 禁止把 AI 修改直接写入文件而不允许用户审阅。
12. 禁止在日志中输出 API 密钥和完整文档内容。
13. 禁止为了模仿 Typora 而复制其品牌、图标、专有资源或非公开实现。
14. 禁止在 V1 引入账户、云同步、协作和插件市场拖慢核心编辑器交付。
15. 禁止正式发布包启用调试自动化桥接或宽松 CSP。

# 34. 深度评审结论

本 PRD 经终审后确认：

- 以 Paperling 为主底座比以 MarkEdit 为主底座更符合三平台目标；
- MarkEdit 仅作为编辑器工程参考，避免引入 Swift/AppKit 平台耦合；
- V1 的核心不是复制 Typora 界面，而是实现稳定、无损、可回退的单编辑区实时渲染；
- 最大技术风险是 IME、光标、撤销历史、双状态和复杂 Widget 性能，因此均已设置发布门槛；
- Linux PDF 采用系统打印到 PDF 作为 V1 可接受基线，避免引入 Chromium；
- Live 模式须经历 Beta 门槛后再设为默认；
- 数据完整性和 Markdown 无编辑往返高于视觉效果；
- Codex 必须按阶段执行，不允许一次性推倒重写；
- 国际化必须在编辑器大规模 UI 重构之前完成基础设施；
- 简体中文是首次安装默认语言，英语为完整支持和正式回退语言；
- UI 语言、文档语言和 AI 输出语言必须相互独立。

终审状态：**通过；V1.2 替代 V1.1，作为 mdtxt 0.1.0 开发与验收基线。**

```text
产品名称：mdtxt
```

最终方向不是复制 Typora，而是实现一款：

> 以开放 Markdown 文件为中心、默认中文、完整支持英语、具备单编辑区实时渲染、跨平台、轻量、本地优先、可回退源码模式的现代 Markdown 编辑器：mdtxt。
